# 需求总览

更新时间：2026-03-26 10:35:00 CST

## 当前需求

### 需求 A

- 名称：提现接口及逻辑改造
- 核心点：
  - 新增提现限制校验模块
  - 预留 `enableWithDraw` 字段能力
  - 新增 `needRuleCheck` 字段控制是否按平台规则限制提现

### 需求 B

- 名称：新增扣款接口
- 核心点：
  - 复用消费三张表结构
  - 新增 `D` 扣款交易类型
  - 支持 `useFrozen=true/false` 两种模式
  - `useFrozen=true` 时复用原冻结额度池能力
  - 是完整扣款交易，仍需调用 `front` 消费接口
  - 仅支持 `04`，不支持分账
  - 支持沿用消费异步上账能力

### 需求 C

- 名称：原冻结交易改造
- 核心点：
  - 原冻结金额支持多笔部分解冻
  - 原冻结金额支持多笔冻结扣款
  - 累计使用额度达到原冻结金额后关闭原冻结
  - 对外保留独立冻结/解冻 Web API

### 需求 D

- 名称：划付改造
- 核心点：
  - 保留原划付模式
  - 新增定时任务划付模式和模式开关
  - 新模式可在划付后写入冻结记录，供后续扣款/解冻使用
  - `02` 模式沿用 `01` 的划付明细新增接口
  - `02` 模式下 `batchNo` 不强校验
  - `isFrozen=true` 时，一笔划付成功对应一笔独立原冻结 `F`

## 依赖关系判断

### 有前后依赖

1. `需求 C -> 需求 B`
   - `需求 B` 中 `useFrozen=true` 的冻结扣款逻辑依赖 `需求 C` 提供的原冻结额度池能力

### 弱依赖

1. `需求 C -> 需求 D`
   - `需求 D` 新模式下如果开启划付冻结，则应复用 `需求 C` 的冻结额度池模型

### 可并行讨论

1. `需求 A`
   - 与冻结/扣款/划付能力无强依赖，可独立设计
2. `需求 D`
   - 主体流程可独立讨论，但涉及冻结落账部分需与 `需求 C` 对齐

## 推荐推进顺序

1. `需求 C` 原冻结交易改造
2. `需求 B` 新增扣款接口
3. `需求 D` 划付改造
4. `需求 A` 提现接口改造

## 当前结论

- `需求 C` 应沉淀为“原冻结额度池能力”
- `需求 B`、`需求 C` 对外都保留独立 Web API
- `B`、`C` 之间复用的是内部冻结额度控制能力，不是接口层
- `需求 C` 的公共能力层只负责 `trans_frozen_t`，不处理账户金额和 `trans_acct_frozen_change_detail_t`
- `需求 C` 是内部纯解冻业务；`需求 B` 是完整扣款交易，`useFrozen=true` 时消费已冻结金额
- `需求 D` 在 `02 + isFrozen=true` 时，是 `需求 C/B` 的原冻结来源生产方
- `B/C/D` 都需要优先校验冻结相关业务 `transNo` 唯一性，但唯一性校验对象是“本次业务流水号”，不是“原冻结流水号只能使用一次”
