# 需求 B 讨论日志

## 2026-03-25 15:42:00 CST

### 讨论背景

在完成 `需求 C` 的原冻结额度池边界收口后，继续分析 `需求 B` 与 `需求 C` 的复用关系，并明确 `需求 B` 的交易定位。

### 已确认事实

1. `需求 B` 虽然逻辑复杂度低于普通消费，但本质上仍是一笔完整扣款交易
2. `需求 B` 仍沿用消费三张表结构
3. `需求 B` 银行侧仍需调用 `front` 消费接口
4. `需求 B` 只支持 `04`
5. `需求 B` 不支持分账

### 用户确认的关键边界

- `需求 C` 是内部纯解冻业务
- `需求 B` 是“解冻额度池控制 + 扣款交易”的组合业务
- 公共抽象层只控制原冻结可用额度
- 账户实际操作放在各自业务线 after 中处理

### 已确认的并发与交易结论

- 同一卡号交易入口本身会加 Redis 锁
- 同一卡号同一时间只允许一笔交易处理
- 因此 `useFrozen=true` 不需要再额外引入一层“预冻结/预解冻预占用”
- 在持锁区间内完成 `oldFrozenTransNo` 的可用额度校验即可

### 已确认的 useFrozen=true 语义

- 消费的是“已冻结金额”
- 不是重新执行普通消费的“冻结 -> 调银行 -> 解冻”链路
- 交易成功后：
  - 写 `D` 使用记录
  - 更新原冻结主记录累计已使用额度
  - 执行账户扣余额、减冻结金额
- 交易失败后：
  - 不写 `D`
  - 不消耗原冻结额度
  - 不变更账户余额和冻结金额

### 已确认的后处理方向

- `需求 B` 支持沿用消费异步上账能力
- 需要参考现有消费 after 和异步上账处理机制
