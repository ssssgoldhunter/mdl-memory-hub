# 需求 B 摘要

更新时间：2026-03-26 16:55:00 CST

## 需求名称

新增扣款接口

## 业务定位

`需求 B` 不是内部能力接口，而是一笔完整扣款交易：

- 仍沿用消费三张表结构保存交易数据
- 银行侧仍调用 `front` 的消费接口
- 交易字段要求尽量与消费保持一致，保证查询、对账、报表兼容

## 已确认结论

### 1. 基本交易边界

- 只支持 `04`
- 不支持分账
- 新建独立请求对象
- 字段尽量复用消费面密对象命名
- 仅保留付款卡、收款卡两个卡号字段
- 仍保留完整交易流程，不简化为纯账户处理接口

### 2. 两种模式

- `useFrozen=false`
  - 走普通 `04` 扣款路径
  - 只校验可用余额
  - 交易过程参考正常消费，保留原有冻结/解冻处理语义
- `useFrozen=true`
  - 消费已冻结金额
  - 必须传 `oldFrozenTransNo`
  - 复用 `需求 C` 沉淀出的原冻结额度池能力

### 3. 与需求 C 的关系

- `需求 B` 不复用 `需求 C` 的 Web API
- `需求 B` 只复用内部冻结额度控制能力
- 公共能力层只负责 `trans_frozen_t`
- 账户金额处理、冻结明细、交易 after 均由 `需求 B` 自己负责

### 4. useFrozen=true 的交易语义

- 这是“消费已冻结金额”，不是“再走一轮普通消费冻结/解冻”
- 交易前在付款卡 Redis 锁保护下完成原冻结额度校验
- 交易成功后才正式：
  - 在 `trans_frozen_t` 写 `D` 记录
  - 更新原冻结主记录累计已使用额度
  - 在 `B after` 中执行账户扣减和冻结金额减少
- 交易失败后：
  - 不写 `D`
  - 不消耗原冻结额度
  - 不变更账户余额和冻结金额

补充唯一性规则：

- `需求 B` 需要优先校验扣款主流水号 `trans_no` 唯一
- 因为 `D.trans_no` 直接使用 `需求 B` 的扣款主流水号
- 同一笔扣款不能重复消耗同一笔原冻结额度

### 5. 账户处理结论

`useFrozen=true` 时，交易成功后付款卡账户侧应同时：

- 减少 `balance`
- 减少 `frozenAmt`

不再额外走普通消费的“冻结 -> 解冻”路径。

### 6. 冻结明细与交易类型

- `trans_frozen_t` 新增 `D` 记录
- `D.trans_no` 直接使用扣款主流水号
- `D.org_trans_no` 指向原冻结流水号
- `trans_acct_frozen_change_detail_t` 由 `B after` 写入，记录本次扣款导致的冻结金额前后变化

### 7. 异步上账

- `需求 B` 支持沿用消费的异步上账能力
- 仅补 `D + 04` 的最小支持，不扩展其他交易类型
- 设计上应与现有消费 after 逻辑保持一致，尽量复用已有异步上账模型和收款侧锁处理方式

### 8. 清结算通知与 front 边界

- `需求 B` 不在主交易成功后立即通知清结算
- 触发时机为：
  - 异步上账完成后
  - 或补偿处理完成后
- 业务系统负责组装 `B` 的业务 DTO 并投递到 `front`
- `front` 负责转成清结算请求报文、发起调用与重发
- `front` 对 `B` 通知采用 24 小时内持续重发，超过 24 小时登记失败
- `需求 B` 在 `front` 中需使用独立通知类型，但处理实现优先复用现有消费通知处理能力
