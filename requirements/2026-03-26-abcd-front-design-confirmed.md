## 1. 文档目的

## 2. 总体结论

### 2.1 清结算对接统一边界

- 清结算相关对接统一收口到 `front`
- `A` 通过 `front` 提供的查询能力完成提现前规则校验
- `A` 的提现结果通知继续沿用现有 `task -> front -> HTTP` 模式
- `B`、`D` 的清结算通知统一通过 `front` 发起
- `C` 不走异步通知模型，而是通过独立 Web API / service 接口直接提供冻结、解冻能力

### 2.2 清结算调用方式

- `front -> 清结算` 第一版调用方式暂定 `HTTP/Feign`
- 当前没有最终的清结算成功返回规范
- 设计上先保留“明确成功才算成功”的原则
- 具体返回码、返回字段待清结算接口最终定义

### 2.3 重发机制

- `front` 对 `B`、`D` 的清结算通知采用持续重发机制
- 重发窗口为 `24` 小时
- `24` 小时内未成功则继续重发
- 超过 `24` 小时仍未成功，则登记失败

## 3. 需求 A 结论

### 3.1 基本口径

- `enableWithDraw` 取消，不做预留
- `needRuleCheck` 作为 Web 对外提现接口入参
- `needRuleCheck` 的语义是：本次提现是否需要执行新增规则校验

### 3.2 提现前规则校验

- 提现前规则校验发生在：
  - 基础参数校验通过后
  - 正式发起提现前
- 该能力通过新增校验模块实现
- 该模块放在提现 LiteFlow 链路中，作为独立节点
- 是否执行由 `needRuleCheck` 控制

### 3.3 清结算查询规则

- `needRuleCheck=true` 时，调用清结算查询接口
- 查询维度只按卡号
- 当前先按“待清结算提供查询 API”设计
- 查询逻辑写在新增校验模块中

### 3.4 拦截口径

- 只要查到该卡号存在扣款失败数据
- 直接拦截提现
- 不提现
- 不额外区分失败原因
- 当前不增加时间窗口限制

### 3.5 提现结果处理与通知

- 提现结果处理仍为异步模式
- 提现接口受理后，不同步等待最终提现结果
- 由 task 后续查单、更新状态
- 状态明确后，由 task 通过 `front` 发起提现结果通知
- 现有链路继续沿用：
  - `task -> front -> HTTP`

## 4. 需求 B 结论

### 4.1 业务定位

- `需求 B` 不是内部能力接口，而是一笔完整扣款交易
- 仍沿用消费三张表结构保存交易数据
- 银行侧仍调用 `front` 的消费接口

### 4.2 请求对象边界

- 新建独立请求对象
- 字段命名尽量复用现有面密消费对象
- 不要分账字段
- 不要多账户类型能力
- 只支持 `04`
- 仅保留两个卡号字段：
  - 付款卡
  - 收款卡
- 其他能复用面密消费字段的尽量复用

### 4.3 两种模式

#### `useFrozen=false`

- 完全不进入原冻结额度池能力
- 只校验可用余额
- 走普通 `04` 扣款路径
- 交易过程参考正常消费
- 保留原有冻结 / 解冻处理语义

#### `useFrozen=true`

- 表示消费已冻结金额
- `oldFrozenTransNo` 必传
- 复用 `需求 C` 的原冻结额度池能力
- 不再额外走普通消费那套“冻结 -> 解冻”语义

### 4.4 after 处理边界

- `useFrozen=false`：
  - 复用普通 `04` 扣款 after
- `useFrozen=true`：
  - 走独立 `B after`
  - 负责：
    - 扣 `balance`
    - 减 `frozenAmt`
    - 写冻结金额变动明细
    - 衔接异步上账

### 4.5 异步上账

- `需求 B` 继续沿用现有异步上账能力
- 只补 `D + 04` 的最小支持
- 不扩展其他交易类型

### 4.6 清结算通知触发时机

- `需求 B` 不在主交易成功后立即通知清结算
- 触发时机为：
  - 异步上账完成后
  - 或补偿处理完成后

### 4.7 与 front 的职责边界

- 业务系统负责组装 `B` 的业务 DTO 并投递到 `front`
- `front` 负责：
  - 转成清结算请求报文
  - 发起调用
  - 执行重发
- `需求 B` 在 `front` 中需使用独立通知类型
- 处理实现优先复用现有消费通知处理能力

## 5. 需求 C 结论

### 5.1 能力定位

- `需求 C` 已收口为“原冻结额度池 + 独立冻结/解冻能力”模型
- `需求 C` 对外需要提供独立 Web API
- 外部用户通过 Web API 调用冻结、解冻功能

### 5.2 清结算接入边界

- `需求 C` 同时保留独立 service 接口
- 清结算通过 `C` 的 service 接口实现冻结、解冻
- `需求 C` 不走异步通知模型

### 5.3 与 B/D 的关系

- `需求 B`、`需求 D` 复用的是 `C` 的内部冻结额度控制能力
- `C` 的冻结表、解冻明细中的 `trans_no` 继续使用业务主流水
- 不为清结算统一划付流水额外新增字段

## 6. 需求 D 结论

### 6.1 基本模式

- 保留原 `01` 划付模式
- 新增 `02` 定时任务划付模式
- `02` 模式沿用 `01` 的划付明细新增接口

### 6.2 `02` 模式接口受理阶段

- `web/api` 入口只负责受理并落待处理记录
- 接口阶段不做：
  - 实际划付
  - 账户变动
  - 冻结金额处理
  - 冻结主记录写入

### 6.3 `02` 模式 task 阶段

- 真正的划付执行由 task 完成
- task 负责：
  - 实际内部上下账
  - 状态推进
  - 必要的失败补偿处理
- 能复用现有 `01` 划付能力的尽量复用
- `02` 特有部分单独拆分

### 6.4 `isFrozen=true` 场景

- 一笔划付成功对应一笔独立原冻结 `F`
- 冻结主记录写入时机为：
  - 账户实际完成划付后
  - 不在接口受理时预写
- `F.trans_no` 直接使用划付流水号
- 后续 `B/C` 的 `oldFrozenTransNo` 就是该划付流水号

### 6.5 清结算通知触发时机

- `需求 D` 不在接口受理成功后立即通知清结算
- 触发时机为：
  - task 实际划付成功后
  - 或补偿处理完成后

### 6.6 与 front 的职责边界

- 业务系统负责组装 `D` 的业务 DTO 并投递到 `front`
- `front` 负责：
  - 转成清结算请求报文
  - 发起调用
  - 执行重发
- `需求 D` 在 `front` 中需使用独立通知类型
- 处理实现优先复用现有转账通知处理能力

## 7. front 对接结论

### 7.1 统一边界

- 清结算对接统一收口到 `front`
- `A` 走查询能力
- `B`、`D` 走异步通知能力

### 7.2 通知类型与处理实现

- `B`、`D` 的通知类型分开
- 但 consumer / 处理类不强制全部新拆
- 能复用现有消费、转账通知处理类的优先复用
- 也就是：
  - 消息语义分开
  - 处理实现尽量复用

### 7.3 24 小时重发

- `front` 对 `B`、`D` 的清结算通知在 `24` 小时内持续重发
- 超过 `24` 小时仍未成功，登记失败
- 第一版需要在 `front` 现有框架上补齐该规则，不能只保留原有限次重发能力
