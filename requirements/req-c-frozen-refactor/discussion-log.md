## 2026-03-25 11:14:45 CST

### 讨论背景

用户提出“原冻结交易改造”需求，希望结合现有代码框架判断依赖关系，并逐步完善需求细节，同时把讨论结果沉淀到记忆体项目。

### 已确认事实

1. 当前系统存在独立冻结/解冻链路
2. `trans_frozen_t` 表结构中已经存在以下字段：
   - `frozen_flag`
   - `allow_flag`
   - `freezable_amt`
   - `freezed_amt`
3. 当前代码逻辑明确限制：
   - 一笔冻结只能完整解冻一次
   - 不支持部分解冻
4. `trans_acct_frozen_change_detail_t` 当前已被报表查询直接使用

### 用户确认的业务目标

- 同一笔原冻结金额，允许多笔解冻
- 同一笔原冻结金额，允许多笔冻结扣款
- 多笔解冻与多笔冻结扣款可以混用
- 两类动作累计消耗同一笔原冻结额度
- 累计/使用的冻结金额等于原冻结金额后，原冻结关闭
- 这些使用动作均不可回滚

### 用户确认的接口边界

- `需求 C` 需要独立的 Web API，供外部用户做冻结/解冻操作
- `需求 B` 需要独立的 Web API，供外部用户做扣款操作
- `B` 与 `C` 之间共享的是内部通用能力，而不是接口层

### 用户确认的内部复用方式

- 通用的是“冻结额度控制阶段”
- 它只是业务流程中的一个阶段
- `需求 C` 的解冻流程会使用它
- `需求 B` 的冻结扣款流程也会使用它
- 账户变动放在各自业务的 after 或业务节点中处理

### 用户确认的落表边界

#### 通用冻结额度控制阶段

- 只负责 `trans_frozen_t`

#### 各自业务节点

- 负责 `trans_acct_frozen_change_detail_t`
- 该表表达的是账户冻结金额变动前后关系

### 用户确认的 trans_frozen_t 规则

- 原始冻结记录类型：`F`
- 解冻记录类型：`UF`
- 冻结扣款记录类型：`D`
- `D` 类型记录直接使用 `需求 B` 的主交易流水号作为 `trans_no`
- `UF` 类型记录直接使用本次解冻请求自己的 `trans_no`
- 两类记录都通过 `org_trans_no` 关联原始冻结流水号

### 当前设计结论

- `需求 C` 应沉淀为“原冻结额度池能力”
- 该能力用于控制原冻结额度是否可继续使用及使用后的状态变化
- `需求 B` 是该能力的一个复用方，但仍保留自己的独立业务接口和业务记账逻辑

## 2026-03-25 11:14:45 CST - DDL补充确认

### 用户提供的真实表结构

用户补充了以下两张表的 DDL：

- `catering_his_mdl.trans_frozen_t`
- `catering_his_mdl.trans_acct_frozen_change_detail_t`

### 基于 DDL 的新增确认

#### 1. trans_frozen_t 可直接承载 `F / UF / D`

- `frozen_type` 为 `char(2)`
- 当前注释虽只写了 `F-冻结 UF-解冻`
- 但字段长度足够承载 `D` 扣款类型

#### 2. trans_frozen_t 的唯一约束与当前设计兼容

- 唯一键：`udx_cardcode_transno (card_code, trans_no)`
- 结论：
  - 每次部分解冻必须有自己独立的 `trans_no`
  - 每次冻结扣款也必须有自己独立的 `trans_no`
  - `B` 的扣款主流水号可直接作为 `D` 类型冻结流水记录的 `trans_no`

#### 3. trans_frozen_t 缺少 org_trans_no 索引

- 当前存在 `idx_trans_no`
- 但不存在 `org_trans_no` 相关索引
- 按新模型，后续会频繁按“原冻结流水号”查询所有 `UF / D` 使用记录
- 后续设计中建议评估新增 `org_trans_no` 相关索引

#### 4. trans_frozen_t 字段语义需要升级

- `frozen_amt`
  - 继续表示“本条记录交易金额”
- `freezed_amt`
  - 在原冻结主记录上用于表示原冻结总金额
- `freezable_amt`
  - 现有注释为“已解冻金额”
  - 在新模型下应升级理解为“原冻结已累计使用金额”
  - 累计使用应包含：
    - `UF` 部分解冻
    - `D` 冻结扣款

#### 5. trans_acct_frozen_change_detail_t 的定位与用户要求一致

- 该表有完整的前后冻结金额字段：
  - `trans_frozen_amt`
  - `org_frozen_amt`
  - `frozen_amt`
- 因此它更适合作为“账户冻结金额变动前后关系明细”
- 不应作为通用冻结额度控制层的主控制台账
- 仍由各自业务节点负责写入更合理

#### 6. trans_acct_frozen_change_detail_t 也支持 `D`

- `trans_type` 为 `char(2)`，足够承载 `D`
- `frozen_type` 为 `char(2)`，当前注释写 `F / UF`
- 若业务需要，也可扩展记录扣款场景对应的冻结操作语义

### 当前风险提醒

- `trans_frozen_t.allow_flag` 数据库注释为：
  - `0-允许解冻`
  - `1-不允许解冻`
- 但当前代码使用方式疑似与注释存在语义反转风险
- 后续设计或实现前，必须统一：
  - 数据库注释
  - 常量定义
  - 代码判断逻辑

## 2026-03-25 11:14:45 CST - 字段语义确认

### 用户确认的核心字段定义

- `freezed_amt`
  - 原冻结总金额
- `freezable_amt`
  - 原冻结累计已使用金额
- `allow_flag`
  - `0` 可继续使用原冻结额度
  - `1` 不可继续使用原冻结额度

### 用户确认的记录类型规则

#### 原冻结主记录 `F`

- 作为原冻结额度池主记录
- `trans_no` 使用原冻结流水号

#### 解冻使用记录 `UF`

- 每次解冻使用独立 `trans_no`
- `org_trans_no` 关联原冻结流水号

#### 冻结扣款使用记录 `D`

- `trans_no` 直接使用 `需求 B` 扣款主流水号
- `org_trans_no` 关联原冻结流水号

### 当前设计共识

- 核心控制不依赖 `frozen_flag`
- 核心控制依赖：
  - `freezed_amt`
  - `freezable_amt`
  - `allow_flag`

## 2026-03-25 11:14:45 CST - 批量解冻流程结论

### 当前内部真实行为判断

基于现有实现分析：

- 当前批量解冻按明细顺序逐笔执行
- 每笔成功后已真实落库并更新相关数据
- 当前链路未看到明确事务包裹
- 若中途失败，接口层可能整批返回失败，但前面成功明细并不会自动回滚

### 用户确认的新方向

内部流程需要同步改造，不能只改外部返回结构：

- 失败明细不应中断整批处理
- 批量解冻应改造成：
  - 全量预校验
  - 校验通过的逐笔执行
  - 明细失败不终止后续执行
  - 最终返回成功列表和失败列表

### 用户确认的返回方向

- 批量解冻返回拆分为：
  - `successList`
  - `failList`
- 不使用“同一列表加状态字段”的模式
