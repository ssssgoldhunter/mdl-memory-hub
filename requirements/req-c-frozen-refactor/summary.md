# 需求 C 摘要

更新时间：2026-03-26 16:55:00 CST

## 需求名称

原冻结交易改造

## 业务目标

将现有“一笔冻结只允许一次完整解冻”的模型，改造成“原冻结额度池”模型：

- 同一笔原冻结金额允许多笔部分解冻
- 同一笔原冻结金额允许多笔冻结扣款
- 部分解冻与冻结扣款可以共同消耗同一笔原冻结额度
- 累计使用额度达到原冻结金额后，该原冻结关闭，不允许继续使用

## 已确认结论

### 1. 对外能力边界

- `需求 C` 需要提供独立 Web API
- 该 API 用于外部用户发起冻结、解冻操作
- `需求 C` 同时保留独立 service 接口，供清结算侧调用冻结、解冻能力

### 2. 与需求 B 的关系

- `需求 B` 也提供独立 Web API
- `B` 不直接依赖 `C` 的外部 API
- `B`、`C` 之间共用的是内部“冻结额度控制能力”

### 3. 内部能力边界

内部需要抽出“冻结额度控制阶段”，它是业务流程中的一个可复用阶段：

- `需求 C` 的解冻流程会调用它
- `需求 B` 的冻结扣款流程会调用它

### 4. 通用能力负责范围

通用冻结额度控制能力负责：

- `trans_frozen_t`

它只负责：

- 查询原冻结记录
- 校验原冻结是否允许继续使用
- 计算剩余可用冻结金额
- 校验本次使用金额
- 记录本次冻结额度使用流水
- 更新原冻结累计已使用额度与关闭状态

补充边界：

- 公共能力层不负责子账户 `balance`、`realBalance`、`frozenAmt` 更新
- 公共能力层不负责 `trans_acct_frozen_change_detail_t`
- 公共能力层只负责“原冻结额度是否还能被使用”和“使用后的主记录状态变化”

### 5. 非通用能力负责范围

`trans_acct_frozen_change_detail_t` 不由通用冻结额度控制能力负责。

该表由各自业务节点负责写入，用于表达账户冻结金额的前后变动关系：

- `需求 C` 的业务节点负责写解冻对应的账户冻结金额变动明细
- `需求 B` 的业务节点负责写冻结扣款对应的账户冻结金额变动明细

账户处理边界补充：

- `需求 C` 是独立冻结/解冻业务，账户侧由 `C after` 负责处理冻结金额增减
- `需求 B` 是完整扣款交易，账户侧由 `B after` 负责处理扣余额、减冻结金额和异步上账衔接

### 6. trans_frozen_t 类型规则

- 原始冻结记录：`F`
- 解冻记录：`UF`
- 冻结扣款记录：直接使用 `需求 B` 的交易类型 `D`

### 7. trans_frozen_t 流水号规则

- 每次解冻都有自己的 `trans_no`
- 每次冻结扣款都有自己的 `trans_no`
- `org_trans_no` 统一指向原始冻结流水号
- `需求 B` 写入 `trans_frozen_t` 时，`trans_no` 直接使用扣款主流水号，形成强关联

补充唯一性规则：

- `需求 C` 必须优先校验“本次解冻请求 `trans_no` 唯一”
- 不能再沿用“原冻结流水号只能出现一次解冻记录”的旧规则
- 同一个 `org_trans_no` 允许存在多条 `UF`

### 8. trans_frozen_t 字段核心语义

- `freezed_amt`
  - 原冻结总金额
- `freezable_amt`
  - 原冻结累计已使用金额
  - 统计口径包含：
    - `UF` 部分解冻
    - `D` 冻结扣款
- `allow_flag`
  - `0`：允许继续使用原冻结额度
  - `1`：不允许继续使用原冻结额度

### 9. trans_frozen_t 三类记录规则

#### 原冻结主记录 `F`

- `frozen_type = F`
- `trans_no = 原冻结流水号`
- `org_trans_no = null`
- `frozen_amt = 本次原冻结金额`
- `freezed_amt = 原冻结总金额`
- `freezable_amt = 0`
- `allow_flag = 0`
- `status = S`

#### 解冻使用记录 `UF`

- `frozen_type = UF`
- `trans_no = 本次解冻流水号`
- `org_trans_no = 原冻结流水号`
- `frozen_amt = 本次解冻金额`
- `freezed_amt = 原冻结总金额`
- `freezable_amt = 原冻结累计已使用金额（使用后值）`
- `allow_flag = 使用后是否还能继续使用`
- `status = S`

#### 冻结扣款使用记录 `D`

- `frozen_type = D`
- `trans_no = B 的扣款主流水号`
- `org_trans_no = 原冻结流水号`
- `frozen_amt = 本次冻结扣款金额`
- `freezed_amt = 原冻结总金额`
- `freezable_amt = 原冻结累计已使用金额（使用后值）`
- `allow_flag = 使用后是否还能继续使用`
- `status = S`

### 10. 原冻结主记录更新规则

每发生一笔 `UF` 或 `D`：

- 原 `F` 记录的 `freezable_amt += 本次使用金额`
- 若 `freezed_amt - freezable_amt > 0`
  - `allow_flag = 0`
- 若 `freezed_amt - freezable_amt = 0`
  - `allow_flag = 1`

### 11. 解冻接口返回规则

`/api/ums/catering/trans/scTransAccountUFrozen` 保持接口名不变，但返回结构需要升级：

- 外层增加 `resultStatus`
  - `SUCCESS`
  - `PART_SUCCESS`
  - `FAIL`
- 返回结果拆分为两批：
  - `successList`
  - `failList`

失败列表建议至少返回：

- `transNo`
- `orgTransNo`
- `cardCode`
- `amount`
- `errorCode`
- `errorMsg`

### 12. 解冻接口内部流程改造方向

批量解冻内部流程从“任一失败整链抛错”改为：

1. 全量预校验
2. 对校验通过的明细逐笔执行
3. 单笔失败不终止后续明细执行
4. 最终按真实执行结果组装：
   - `successList`
   - `failList`

### 13. 与需求 B 的交易边界补充

- `需求 B` 不是内部能力接口，而是完整扣款交易
- `需求 B` 仍然沿用消费三张表结构
- `需求 B` 银行侧仍调用 `front` 消费接口
- `需求 B` 仅支持 `04`
- `需求 B` 不支持分账
- `需求 B` 支持沿用消费异步上账能力
- `需求 B` 在 `useFrozen=true` 时不再额外做“普通消费冻结/解冻”
- `需求 B` 在交易成功后才正式写 `D` 记录并消耗原冻结额度

### 14. 异常分层规则

#### 批次级错误

以下错误仍然终止整个请求：

- 请求体为空
- 批次参数非法
- 全局锁获取失败
- 批次初始化失败

#### 明细级错误

以下错误记入 `failList`，不终止后续明细：

- 原冻结不存在
- 原冻结已关闭
- 原冻结额度不足
- 单笔冻结流水落账失败
- 单笔账户冻结金额更新失败

### 15. 旧强校验改造要求

现有解冻链路中存在会阻塞“多次部分解冻”设计的旧逻辑，后续必须移除或改造：

- 旧逻辑问题：
  - 以“原冻结已存在解冻记录”为由直接报错
  - 隐含假设“一笔原冻结只能解冻一次”
- 新逻辑要求：
  - 优先校验“本次解冻业务流水号唯一”
  - 再校验原冻结主记录是否存在、是否允许继续使用、剩余额度是否足够
  - 不再把“原冻结已发生过一次解冻”视为错误

## 当前设计判断

- 现有表结构已经预留了部分解冻相关字段
- 但当前业务逻辑仍限制“一笔冻结只能一次性全额解冻”
- 后续改造重点在于把表结构中的预留能力真正落地为可复用的冻结额度池能力
- 基于真实 DDL 判断：
  - `trans_frozen_t` 可继续作为冻结额度池主台账
  - `trans_acct_frozen_change_detail_t` 更适合作为账户冻结金额变动前后关系明细
  - `trans_frozen_t.freezable_amt` 的业务语义需要从“已解冻金额”升级为“原冻结已累计使用金额”
  - `trans_frozen_t` 后续建议补充 `org_trans_no` 相关索引评估
  - `frozen_flag` 不建议作为核心控制字段，核心控制依赖 `freezed_amt`、`freezable_amt`、`allow_flag`
  - 当前内部批量解冻真实处理并非整批原子事务，后续必须让返回语义与真实落库结果一致
  - `需求 C` 已基本收口为“原冻结额度池 + 独立冻结/解冻能力”模型，后续主要服务于 `需求 B` 和 `需求 D` 复用

## 影响需求

- `需求 B` 的 `useFrozen=true` 依赖本需求提供的能力
- `需求 D` 如果开启划付冻结，也建议复用本需求形成的冻结额度池模型
- `需求 C` 自身不走异步通知清结算模式，而是通过独立 Web API / service 接口直接提供冻结、解冻能力
