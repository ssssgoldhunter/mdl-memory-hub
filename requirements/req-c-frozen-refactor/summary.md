# 需求 C 摘要

更新时间：2026-03-31 10:10:00 CST

## 需求名称

原冻结交易改造

## 业务目标

将现有“一笔冻结只允许一次完整解冻”的模型，改造成“原冻结额度池”模型：

- 同一笔原冻结金额允许多笔部分解冻
- 同一笔原冻结金额允许多笔冻结扣款
- 部分解冻与冻结扣款可以共同消耗同一笔原冻结额度
- 累计使用额度达到原冻结金额后，该原冻结关闭，不允许继续使用

## 已确认结论

### 1. 对外能力边界

- `需求 C` 需要提供独立 Web API
- 该 API 用于外部用户发起冻结、解冻操作
- `需求 C` 同时保留独立 service 接口，供清结算侧调用冻结、解冻能力

### 2. 与需求 B 的关系

- `需求 B` 也提供独立 Web API
- `B` 不直接依赖 `C` 的外部 API
- `B`、`C` 之间共用的是内部“冻结额度控制能力”

### 3. 内部能力边界

内部需要抽出“冻结额度控制阶段”，它是业务流程中的一个可复用阶段：

- `需求 C` 的解冻流程会调用它
- `需求 B` 的冻结扣款流程会调用它

### 4. 通用能力负责范围

通用冻结额度控制能力负责：

- `trans_frozen_t`

它只负责：

- 查询原冻结记录
- 校验原冻结是否允许继续使用
- 计算剩余可用冻结金额
- 校验本次使用金额
- 生成本次 `UF/D` 冻结流水
- 更新原冻结累计已使用额度与关闭状态

补充边界：

- 公共能力层不负责子账户 `balance`、`realBalance`、`frozenAmt` 更新
- 公共能力层不负责 `trans_acct_frozen_change_detail_t`
- 公共能力层只负责“原冻结额度是否还能被使用”、“本次 `UF/D` 流水生成”和“原始 `F` 使用后的主记录状态变化”

### 5. 非通用能力负责范围

`trans_acct_frozen_change_detail_t` 不由通用冻结额度控制能力负责。

该表由各自业务节点负责写入，用于表达账户冻结金额的前后变动关系：

- `需求 C` 的业务节点负责写解冻对应的账户冻结金额变动明细
- `需求 B` 的业务节点负责写冻结扣款对应的账户冻结金额变动明细

账户处理边界补充：

- `需求 C` 是独立冻结/解冻业务，账户侧由 `C` 自己负责处理冻结金额增减
- `需求 B` 是完整扣款交易，账户侧由 `B after` 负责处理扣余额、减冻结金额和收款同步上账

### 6. trans_frozen_t 类型规则

- 原始冻结记录：`F`
- 解冻记录：`UF`
- 冻结扣款记录：直接使用 `需求 B` 的交易类型 `D`

### 7. trans_frozen_t 流水号规则

- 每次解冻都有自己的 `trans_no`
- 每次冻结扣款都有自己的 `trans_no`
- `org_trans_no` 统一指向原始冻结流水号
- `需求 B` 写入 `trans_frozen_t` 时，`trans_no` 直接使用扣款主流水号，形成强关联

补充唯一性规则：

- `需求 C` 必须优先校验“本次解冻请求 `trans_no` 唯一”
- 不能再沿用“原冻结流水号只能出现一次解冻记录”的旧规则
- 同一个 `org_trans_no` 允许存在多条 `UF`

### 8. trans_frozen_t 字段核心语义

- `freezed_amt`
  - 原冻结总金额
- `freezable_amt`
  - 原冻结累计已使用金额
  - 统计口径包含：
    - `UF` 部分解冻
    - `D` 冻结扣款
- `allow_flag`
  - `0`：允许继续使用原冻结额度
  - `1`：不允许继续使用原冻结额度

### 9. 原冻结主记录更新规则

每发生一笔 `UF` 或 `D`：

- 原 `F` 记录的 `freezable_amt += 本次使用金额`
- 若 `freezed_amt - freezable_amt > 0`
  - `allow_flag = 0`
- 若 `freezed_amt - freezable_amt = 0`
  - `allow_flag = 1`

### 10. 与需求 B 的交易边界补充

- `需求 B` 不是内部能力接口，而是完整扣款交易
- `需求 B` 仍然沿用消费三张表结构
- `需求 B` 银行侧仍调用 `front` 消费接口
- `需求 B` 仅支持 `04`
- `需求 B` 不支持分账
- `需求 B` 在 `useFrozen=true` 时不再额外做“普通消费冻结/解冻”
- `需求 B` 在交易成功后才正式写 `D` 记录并消耗原冻结额度
- `需求 B` 引用原冻结时使用 `orgFrozenTransNo`
- `需求 C` 引用原冻结时使用 `orgTransNo`

### 11. 当前设计判断

- 现有表结构已经预留了部分解冻相关字段
- 当前业务逻辑已按“原冻结额度池”方向收口
- 基于真实 DDL 判断：
  - `trans_frozen_t` 继续作为冻结额度池主台账
  - `trans_acct_frozen_change_detail_t` 继续作为账户冻结金额变动前后关系明细
  - `trans_frozen_t.freezable_amt` 的业务语义是“原冻结已累计使用金额”
  - `frozen_flag` 不作为核心控制字段，核心控制依赖 `freezed_amt`、`freezable_amt`、`allow_flag`

## 影响需求

- `需求 B` 的 `useFrozen=true` 依赖本需求提供的能力
- `需求 C` 自身通过独立 Web API / service 接口直接提供冻结、解冻能力
