# 当前实现基线

更新时间：2026-03-31 10:10:00 CST

## 0. 计划归属

- 大计划名称：
  - `mdl定制`
- 当前 `A/B/C/D/front` 需求统一归属 `mdl定制`
- 本文档描述的是 `mdl定制 -> 供应链 ABCD` 当前唯一有效基线

## 1. 总原则

当前这组需求后续一律按下面这套基线继续：

1. `A` 提现规则校验
2. `C` 原冻结额度池能力
3. `B` 扣款主交易
4. `D` 划付模式改造
5. `front` 通知与重发
6. 测试与联调

这里的核心不是“开发顺序”，而是下面几条边界：

- `D` 全部属于“划付域”，不属于原转账链
- `B` 是完整扣款交易，不是内部能力接口
- `C` 是 `B` 复用的原冻结额度池能力
- `front` 负责 `B/D` 的结果通知，不负责 `A` 的提现前查询
- 历史设计稿只用于追溯，不再作为当前开发依据

## 3. C 基线

### 3.1 能力定义

`C` 当前应理解为：

- 原冻结额度池能力
- 独立冻结能力
- 独立解冻能力

### 3.2 核心状态语义

原冻结记录的剩余额度管理依赖：

- `frozenAmt`
- `freezableAmt`
- `allowFlag`

当前已落地的语义是：

- 支持按剩余冻结额度做部分解冻
- 支持按剩余冻结额度做部分冻结扣款
- 全部释放或全部使用后，`allowFlag` 关闭

### 3.3 公共冻结池边界

公共冻结池操作只负责：

- 查询原冻结 `F`
- 校验原冻结是否允许继续使用
- 计算本次可用剩余额度
- 生成本次 `UF/D` 记录
- 更新原始 `F` 的 `freezableAmt/allowFlag`

公共冻结池不负责：

- 子账户 `balance` / `realBalance` / `frozenAmt` 更新
- `trans_acct_frozen_change_detail_t`

这些仍然在各自业务代码中处理：

- `C` 自己负责解冻侧冻结金额变动明细
- `B` 自己负责冻结扣款侧冻结金额变动明细

### 3.4 原冻结流水号口径

- `C` 解冻只使用：
  - `orgTransNo`
- `B` 冻结扣款使用：
  - `orgFrozenTransNo`
- 不再引入额外“旧冻结流水号”概念

## 4. B 基线

### 4.1 入口与核心链边界

- 对外入口可复用现有消费 controller / service
- 但核心交易链必须与普通消费区分
- `B` 主交易类型是 `D`
- `B` 仍沿用消费三张表结构

### 4.2 允许复用

- 通用校验
- 通用上下文
- 通知基础设施
- 原 `TransSlot`
- 消费三张表结构

### 4.3 不允许混用

- 普通消费主交易链
- 普通消费 after
- 原转账链
- 普通消费主交易类型 `P`
- 普通消费账户变动明细类型 `PC/PR`

### 4.4 两种模式

#### `useFrozen=false`

- 走普通扣款模型
- 只支持 `04`
- 正式交易前先校验 `04` 可用余额
- 正式交易前先做冻结金额占用
- 账户变动前先解冻
- 再同步更新付款卡、收款卡
- 这里只体现冻结金额变动明细
- 不进入冻结池主记录模型

#### `useFrozen=true`

- 走冻结扣款模型
- 只支持 `04`
- 直接消费原冻结额度
- 不再额外做普通冻结 / 解冻
- 校验的是：
  - 原冻结剩余额度
  - 账户当前冻结金额
  - 账户总冻结金额与当前原冻结剩余额度的一致性保护
- 使用 `C` 的冻结池控制能力
- 同步更新付款卡、收款卡

### 4.5 锁与账户变更边界

- `B` 在 service 入口层先锁付款卡、收款卡
- 两把锁都成功，才进入正式交易
- 两个 after 都是同步账户变更
- 不再走收款异步上账

### 4.6 交易与明细类型

- 主交易类型：
  - `D`
- 账户变动明细类型：
  - 付款方 `DC`
  - 收款方 `DR`
- `useFrozen=true` 的冻结金额变动明细：
  - `frozenType = D`
  - `transType = D`

## 5. D 基线

### 5.1 模式开关

- 优先读取参数表：
  - `TRANSFER_MODE_SWITCH`
- 开关只在“划付域”生效
- 不再侵入原转账入口、原转账链、原转账 service

### 5.2 01 模式

- 保持原有划付逻辑
- 原有 `01` 的建批、触发、task 执行链不再继续为 `02` 改造
- `01` 原有上下账逻辑保持不动

### 5.3 02 模式

- `02` 是新增的一套划付模式，不是转账模式
- `02` 接口阶段只负责受理并落划付明细
- `02` 不再自动创建内部批次
- `02` 不依赖 `batchNo` 启动
- `02` 只参考 / 复用 `01` 的划付明细落地方式
- `02` 的真正启动入口在 `fund-catering-task`
- `02` task 直接翻页扫描待处理明细表
- `02` task 逐条调用单接口 `processDetail02`
- `processDetail02` 内部一次性完成：
  - 付款侧处理
  - 收款侧处理
  - 状态推进
  - `isFrozen` 冻结写入
  - 成功通知发送
- `01` 原有上下账接口不为 `02` 复用，不为 `02` 改造

### 5.4 02 模式禁止再回到的错误概念

- 不再对外暴露：
  - `registerMode02Detail`
  - `batchRegisterMode02Detail`
- 不再把 `02` 理解成“完整复用 `01` 的外部建批/触发接口”
- 不再把 `02` 的开关判断继续侵入 `01` 原有入口
- 不再在 `data-batch` 中实现 `02` 启动
- 不再把 `02` 写成“两接口串行执行”的模式

### 5.5 冻结相关语义

- `useFrozen=true`
  - task 执行时消费原冻结额度
- `isFrozen=true`
  - 收款卡上账成功后立即写新的原冻结 `F`
  - 账户冻结金额更新继续复用 `createFrozenDetail(...)`
- 新的 `F` 可继续被 `B/C` 使用
- `D` 不再使用 `oldFrozenTransNo`
- 后续引用这笔冻结时：
  - `C` 解冻使用 `orgTransNo`
  - `B` 冻结扣款使用 `orgFrozenTransNo`
