# 当前实现基线

更新时间：2026-03-27 19:05:00 CST

## 1. 总原则

当前这组需求后续一律按下面这套基线继续：

1. `A` 提现规则校验
2. `C` 通用冻结额度管理能力
3. `B` 扣款主交易
4. `D` 划付模式改造
5. `front` 通知与重发
6. 测试与联调

这里的重点不是“开发顺序”本身，而是：

- `C` 是 `B/D` 的冻结额度基础能力
- `B` 核心交易链独立
- `D02` 只复用 `01` 中间那段批量明细上传
- `front` 负责 `B/D` 通知，不负责 `A` 的提现前查询

## 2. A 基线

### 2.1 当前正确口径

- `A` 的新增规则校验放在提现 `LiteFlow`
- 独立节点名：
  - `withDrawRuleCheck`
- 位置：
  - 常规 check 之后
  - 正式提现前

### 2.2 当前行为

- `needRuleCheck=false`
  - 直接跳过节点
- `needRuleCheck=true`
  - 进入节点
  - 当前只记录日志和 `TODO`
  - 暂不真实拦截

### 2.3 当前未完成项

- 清结算查询 API 还没提供
- 所以还没接真实查询和真实拦截

### 2.4 front 边界

- `front` 不承担 `A` 的提现前规则查询
- `front` 在 `A` 中只承担提现结果通知

## 3. C 基线

### 3.1 能力定义

`C` 当前应理解为：

- 通用冻结额度管理能力
- 独立冻结能力
- 独立解冻能力

### 3.2 核心状态语义

原冻结记录的剩余额度管理依赖：

- `frozenAmt`
- `freezableAmt`
- `allowFlag`

当前已落地的语义是：

- 支持按剩余冻结额度做部分解冻
- 支持按剩余冻结额度做部分冻结扣款
- 全部释放或全部使用后，`allowFlag` 关闭

### 3.3 原冻结流水号口径

- 解冻只使用：
  - `orgTransNo`
- 不再引入额外“旧冻结流水号”概念

## 4. B 基线

### 4.1 入口与核心链边界

- 对外入口可复用现有消费 controller / service
- 但核心交易链必须与普通消费区分

### 4.2 允许复用

- 通用校验
- 通用上下文
- 通知基础设施
- 原 `TransSlot`

### 4.3 不允许混用

- 普通消费主交易链
- 普通消费 after

### 4.4 当前语义

- `transType = P`
- 仅支持 `04`
- `useFrozen=false`
  - 走 `B` 自己的普通扣款链
- `useFrozen=true`
  - 使用 `C` 的冻结额度管理能力
  - 走独立 `B after`

## 5. D 基线

### 5.1 模式开关

- 优先读取参数表：
  - `TRANSFER_MODE_SWITCH`
- 无参数时再回退请求里的 `transferMode`

### 5.2 01 模式

- 保持原有划付逻辑
- 原有 `01` 的建批、触发、task 执行链不再继续为 `02` 改造

### 5.3 02 模式

- `02` 是新增的一套划付模式，不是转账模式
- `02` 对外受理阶段只做：
  - `transTransferTiPre`
  - `batchAddTransferTiData`
- `02` 只参考 / 复用 `01` 中间那段“批量上传明细”
- `02` 的真正启动入口在 task，不走 `01` 现有入口
- `02` task 执行时调用独立的 `processDebit02 / processCredit02`
- `01` 原有 `processDebit / processCredit` 保持不动

### 5.4 02 模式禁止再回到的错误概念

- 不再对外暴露：
  - `registerMode02Detail`
  - `batchRegisterMode02Detail`
- 不再把 `02` 理解成“完整复用 `01` 的外部建批/触发接口”
- 不再把 `02` 的开关判断继续侵入 `01` 原有入口

### 5.5 冻结相关语义

- `useFrozen=true`
  - task 执行时消费原冻结额度
- `isFrozen=true`
  - task 成功后写新的原冻结 `F`
- 新的 `F` 可继续被 `B/C` 使用

## 6. front 基线

### 6.1 A

- 只管提现结果通知
- 不负责提现前查询

### 6.2 B / D

- `B`
  - `TOPIC_NOTIFY_CATERING_DEDUCTION`
- `D`
  - `TOPIC_NOTIFY_CATERING_TRANSFER`

### 6.3 重发

- 继续复用现有 HTTP consumer 框架
- `24` 小时窗口内持续重发
- 超过 `24` 小时不再继续调度
