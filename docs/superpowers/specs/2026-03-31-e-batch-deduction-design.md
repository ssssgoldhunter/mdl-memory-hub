# E 批量扣款设计稿

更新时间：2026-03-31 13:40:00 CST

## 1. 计划归属

- 大计划名称：
  - `mdl定制`
- 当前 `E` 需求是 `mdl定制 -> 供应链 ABCD` 的后续扩展需求
- `E` 不替代 `A/B/C/D`，而是在现有 `B + D02` 基础上新增“批量扣款”执行模型

## 2. 一句话定义

`E` 是“批量版扣款 02 模式”：

- 扣款业务语义参考当前 `B`
- 执行模型参考当前 `D02`
- 代码链路与单笔 `B` 完全隔离
- 接口阶段先落明细，再预落交易骨架
- task 分页扫明细，逐条调用单入口执行

可简写为：

```text
E = 参考 B 的扣款语义 + 参考 D02 的执行模型
```

## 3. 目标与边界

### 3.1 目标

新增一套批量扣款能力，支持：

- 普通扣款
- 冻结扣款

并满足：

- 接口阶段不做真实账务动作
- task 阶段逐条推进真实执行
- 单条执行入口内部完成完整扣款流程

### 3.2 不做的事

`E` 当前不做：

- 修改单笔 `B` 的既有业务语义
- 修改原消费主链
- 修改原转账链
- 按 `batchNo` 整批同步执行
- 接口阶段直接做正式扣款和账户变更

## 4. 与现有需求的关系

### 4.1 参考 B 的部分

`E` 参考 `B` 的扣款业务语义，但不与 `B` 共用同一套执行链：

- 主交易类型仍是 `D`
- 账户变动明细类型仍是：
  - 付款方 `DC`
  - 收款方 `DR`
- 仍然只支持 `04`
- 仍然分两种模式：
  - `useFrozen = N`
  - `useFrozen = Y`

### 4.2 参考 D02 的部分

`E` 参考 `D02` 的执行模型：

- 接口受理阶段只落业务数据
- task 分页扫明细
- task 逐条执行
- 外部只保留一个单条执行入口

### 4.3 参考 C 的部分

当 `useFrozen = Y` 时：

- 继续参考 `C/B` 当前冻结池语义
- 通过 `orgFrozenTransNo` 找原始冻结 `F`
- 生成本次 `D`
- 更新原始 `F`

## 5. 两种扣款模式

### 5.1 普通扣款

条件：

- `useFrozen = N`

语义：

- 走当前 `B useFrozen=false`
- 正式交易前先冻结占用
- 账户变动前先解冻
- 再同步更新付款卡、收款卡

### 5.2 冻结扣款

条件：

- `useFrozen = Y`
- `orgFrozenTransNo` 必填

语义：

- 走当前 `B useFrozen=true`
- 直接消费原冻结额度
- 不走普通冻结/解冻
- 同步更新付款卡、收款卡
- 生成本次 `D`
- 更新原始 `F`

- `trans_deduction_batch_detail`

- `transAmt`
- `payCardCode`
- `receiveCardCode`
- `subAccRange`

  - 默认 `N`
- `orgFrozenTransNo`
  - 仅 `useFrozen = Y` 时必填
- `frontStatus`
- `dcStatus`
- `drStatus`
- `status`
- `operatorCode`
- `firstOrgCode`
- `secondOrgCode`
- `merchantCode`
- `terminalCode`
- `userBusinessType`
- `userBusinessCode`
- `userBusinessCommon`
- `userBusinessTags`

### 6.2 字段规则

- `transNo`
  - 由调用方提供
  - 接口阶段必须校验非空和唯一
- `useFrozen`
  - 默认值为 `N`
- `orgFrozenTransNo`
  - `useFrozen = N` 时为空
  - `useFrozen = Y` 时必填

## 7. 阶段 1：接口预落地

### 7.1 总原则

接口阶段不做真实账务动作，只做：

- 校验
- 预落地明细
- 预落地交易骨架

### 7.2 固定顺序

接口阶段顺序固定为：

```text
先落明细 -> 再落交易
```

不允许改成：

- 先落交易再落明细
- 明细和交易乱序插入

### 7.3 预落地要落的表

阶段 1 需要落 4 类表：

1. `trans_deduction_batch_detail`
2. `trans_consume_t`
3. `trans_consume_sub_t`
4. `trans_consume_sub_rec_t`

说明：

- 这是对齐当前单笔 `B` 正式创建交易时的交易骨架结构
- `E` 只是把“正式创建交易骨架”提前到接口阶段
- 不允许自己新定义一套更少的交易骨架结构

### 7.4 各表职责

#### `trans_deduction_batch_detail`

作用：

- 作为 `E` 的任务明细表
- 也是后续 task 的扫描队列

#### `trans_consume_t`

作用：

- 扣款主交易骨架

建议状态：

- `status = P`

#### `trans_consume_sub_t`

作用：

- 扣款子交易骨架

建议状态：

- 与主交易保持处理中语义

#### `trans_consume_sub_rec_t`

作用：

- 扣款子记录 / 收付关系骨架

建议状态：

- `recStatus = P`
- `accChangeStatus = P`

### 7.5 接口阶段不落的表

接口阶段不应该落：

- 账户变动明细表
- 冻结金额变动明细表
- 原冻结主记录表
- 入账分录结果表
- 实际账务结果类表

原则是：

> 接口阶段只落“业务明细 + 交易骨架”，不落“账务结果”。

## 8. 阶段 2：task 扫明细执行

### 8.1 task 形态

`E` 的 task 应采用与 `D02` 相同的执行模型：

- 定时任务
- 分页扫描明细表
- 逐条执行
- 任务级 Redis 锁避免重复并发

### 8.2 扫描对象

扫描表：

- `trans_deduction_batch_detail`

扫描方式：

- 不按 `batchNo` 整批启动
- 直接按待处理明细分页扫描

## 9. 阶段 3：单条执行入口

### 9.1 入口原则

外部只保留一个单条执行入口。

建议命名：

- `processDeductionDetail02`

不建议保留：

- 前后多个串行外部接口
- 普通扣款 / 冻结扣款两个外部分开的执行入口

### 9.2 单条执行入口职责

单条执行入口内部顺序完成：

1. 查询明细
2. 查询预落交易骨架
3. 校验状态，避免重复执行
4. 根据 `useFrozen` 分流
5. 调正式扣款交易
6. 执行付款侧处理
7. 执行收款侧处理
8. 更新状态
9. 记录失败原因或成功结果

## 10. 状态机

### 10.1 字段语义

- `frontStatus`
  - 正式扣款交易调用状态
- `dcStatus`
  - 付款侧处理状态
- `drStatus`
  - 收款侧处理状态
- `status`
  - 整单汇总状态

### 10.2 建议流转

接口受理成功后：

- `frontStatus = I`
- `dcStatus = I`
- `drStatus = I`
- `status = I`

开始正式扣款：

- `frontStatus = P`
- `status = P`

正式扣款成功，开始付款侧处理：

- `frontStatus = S`
- `dcStatus = P`
- `status = P`

付款侧成功，开始收款侧处理：

- `dcStatus = S`
- `drStatus = P`
- `status = P`

收款侧成功：

- `drStatus = S`
- `status = S`

正式扣款失败：

- `frontStatus = F`
- `status = F`

付款侧失败：

- `dcStatus = F`
- `status = F`

收款侧失败：

- `drStatus = F`
- `status = F`

## 11. 锁与账户更新原则

`E` 的锁与账户更新，必须围绕“task 逐条执行”来设计，不能照搬单笔 `B` 的同步入口模型。

`B` 是同步单笔交易：

- 请求进来
- service 入口加锁
- 同步完成整条链

而 `E` 是两阶段模型：

- 接口阶段只预落地
- task 阶段才做真实执行

所以 `E` 的锁设计必须充血到 task 单条执行模型里，不能写成一个过于简化的“沿用 B 口径”。

### 11.1 接口阶段不锁卡

接口受理阶段只负责：

- 校验入参
- 落明细
- 落交易骨架

这一阶段不做真实账务动作，因此：

- 不锁付款卡
- 不锁收款卡

原因是：

- 接口阶段锁卡没有真实执行意义
- 只会放大锁持有时间
- 还会与 task 真执行时机脱节

### 11.2 task 阶段才是真正锁卡阶段

`E` 的锁卡只属于 task 单条执行阶段。

也就是：

- task 扫到某条明细
- 准备进入 `processDeductionDetail02`
- 在真实执行前，才开始获取卡锁

锁卡本质上是“保护单条真实账务执行”，不是“保护预受理”。

### 11.3 单条执行的锁顺序

单条执行入口开始真实处理前，应按固定顺序获取：

1. 付款卡锁
2. 收款卡锁

固定顺序的意义是：

- 降低并发时交叉抢锁的混乱
- 避免不同实现各自定义顺序
- 保持后续故障排查口径一致

### 11.4 单条执行的放行条件

只有在下面条件同时满足时，单条明细才允许继续执行：

- 付款卡锁获取成功
- 收款卡锁获取成功

只拿到一张卡锁，不允许继续。

原因是：

- `E` 的真实账务执行最终要同步处理收付款两边
- 单边先执行、另一边未锁定，会放大并发不一致风险

### 11.5 锁失败时的处理方式

任意一张卡锁获取失败，当前明细不应继续做真实执行。

此时应做的是：

- 当前明细停止继续执行
- 记录本次执行未完成
- 保留为可后续重试的状态，或记录明确失败原因

这里不建议把“锁失败”直接等同于永久业务失败。

更合理的口径是：

- 锁失败本质上更接近“当前并发冲突/资源忙”
- 后续仍可由 task 下一轮再次尝试

### 11.6 解锁时机

单条执行结束后统一释放两张卡锁。

包括：

- 成功释放
- 失败释放
- 抛异常后释放

也就是：

- 解锁必须放在单条执行的最终 `finally` 语义里
- 不能依赖外层 task 批量释放
- 不能把锁状态跨明细持有

### 11.7 账户更新模型

在锁已经拿到的前提下，`E` 的账户更新仍然按当前 `B` 已确认的正确口径执行：

- 收付款账户变更全部同步
- 不走异步收款入账
- 普通扣款与冻结扣款的差异只保留在付款侧语义
- 收款侧统一走同步上账

### 11.8 账户更新参数口径

调用账户更新接口时，必须继续遵守现有底层能力的要求：

- 传增量值
- 不传目标值

也就是：

- 扣减传负数增量
- 增加传正数增量

不能把“更新后余额”直接塞给账户更新接口。

### 11.9 冻结金额处理原则

如果单条执行涉及冻结金额变化：

- 继续复用 `createFrozenDetail(...)`
- 不额外手写一套冻结金额账户更新

这样做的原因是：

- `createFrozenDetail(...)` 已包含现有冻结金额变动能力
- 已复用现有账户查询和更新路径
- 能保持和现有 `B/C/D` 一致的冻结金额处理语义

### 11.10 MAC/CAS 一致性原则

`E` 的账户更新必须继续遵守当前系统已有的 `mac/CAS` 一致性模型。

也就是：

- 真正更新账户时，使用最新账户态
- 继续走现有账户更新服务
- 不自己绕开已有 `mac/CAS` 更新链路

这样可以保证：

- task 多轮执行时账户更新口径一致
- 冻结金额更新和余额更新口径一致
- 不会因为预落地阶段早查过账户，就直接拿旧快照做覆盖更新

### 11.11 本节最终结论

`E` 的锁与账户更新原则可以浓缩成这句：

> `E` 不在接口阶段锁卡，只在 task 单条真实执行前锁付款卡和收款卡；两张卡都锁成功才继续，收付款账户变更全部同步，冻结金额处理继续复用现有能力并保持 `mac/CAS` 一致性。

## 12. 预落地专用 liteflow 设计

`E` 需要新增一条“扣款预落地专用 liteflow”。

该流程只负责：

- 校验请求
- 构建 `B` 交易骨架
- 落明细和交易骨架

该流程不负责：

- 正式调钱包 / 渠道
- 账户变更
- 普通冻结 / 解冻
- 冻结池消费

也就是说：

- 不能直接复用单笔 `B` 当前完整 `deductionTrans` 节点
- 需要从中拆出“仅建骨架”的预落地能力

## 13. 异常与补偿

当前建议第一阶段按下面方式设计：

- 单条失败只影响单条
- 不做整批回滚
- 不做自动整批重试
- 后续可补单条手动补偿入口

需要重点盯住的失败场景：

- 明细已落，交易骨架落失败
- 正式扣款成功，付款侧失败
- 付款侧成功，收款侧失败
- 冻结扣款模式下，原冻结已更新但收款失败

## 14. 当前确认结论

### 14.1 已固定

- `transNo` 必传，由调用方提供
- 明细默认普通扣款：
  - `useFrozen = N`
- `useFrozen = Y` 时：
  - `orgFrozenTransNo` 必填
- 接口阶段顺序固定：
  - 先落明细，再落交易
- 预落交易状态为：
  - `P`
- task 只扫明细表
- task 对外只调一个单条执行入口

### 14.2 实现时禁止再走回的错误方向

- 不要自动生成 `transNo`
- 不要把 `E` 做成整批同步执行
- 不要把 `E` 的 task 改成按 `batchNo` 启动
- 不要把普通扣款和冻结扣款拆成两个外部执行入口
- 不要在接口阶段提前做真实账户变更
- 不要减少交易骨架表数量，与单笔 `B` 脱节
